## 1. 初回アクセス時の権限チェック  〔優先度: 中〕

サーバー起動後、初めてタブにアクセスするとき（または適切なタイミングで）、そのタブで拡張機能の権限が有効か確認するパートを入れる。
権限がない場合、警告をユーザー（agent等）に返す。

- **現状**: タブごとの権限有効性チェックは未実装。`get_sessions` は WS 接続済みタブのみ列挙するため「拡張が動いていないタブ」は気付けない（コンテンツスクリプトが注入されていないタブはセッション自体が存在しない）。`EXT_HELLO`（接続時のバージョン申告）はあるが、タブ単位の権限状態は申告していない。
- 検討: 警告を `get_sessions` / アクション系の `_warnings` に載せる経路がよさそう（`NO_ACTIVE_SESSIONS` と同じ枠組み）。

## 2. 大量タブのレイジー処理  〔優先度: 低〕

一度に大量のタブが同時に追加・読み込まれた場合、レイジー処理（収集を間引く / 後回しにする）を行う。
既に最適化されている場合はスキップ。

- **現状**: アダプティブ収集スケジューラ（`CollectionScheduler`、`extension/background/sw.js`）が two-speed cadence で実装済みだが **デフォルト無効**（`adaptiveCollection.enabled`）。「同時大量タブ」専用のレイジー化ロジックがこの中でカバーされているかは要確認。まずは既存スケジューラの挙動を見てから判断する。

## 3. セッション一覧の優先表示  〔優先度: 高（3+4+5を一体で）〕

`get_sessions` / `GET /api/sessions` で一覧を取得するとき、関連性の高いもの（拡張機能が接続されている実タブ・アクティブタブ・最近更新されたもの）を優先的に先頭に表示する。

- **現状**: 未実装。`read-basic.js` の `get_sessions` ハンドラは `cache.getSessionList()` の結果をそのまま返すだけ（ソートなし）。並べ替えの差し込み先はここ。

## 4. セッション一覧の自動ページング  〔優先度: 高（3+4+5を一体で）〕

セッション数が多い場合、自動でページ分割する（`?page=1` のようにパラメータが自動付与されるイメージ）。

- 最適な分割数について：1ページあたりの平均件数がだいたい揃うようにする？ あるいは固定しきい値（例：20件）を超えたらページ分割？
- agentが望む場合は `page=all` / `page=full` で全件一括表示も可能にする。
- **現状**: 未実装（`get_sessions` の inputSchema は空。`page` 引数なし）。項目3のソートと一緒に実装するのが自然。

## 5. タブ検索  〔優先度: 高（3+4+5を一体で）〕

`get_sessions` に対して以下の検索手段を有効にする：
- ファジー検索（あいまい検索）
- MiniLMでの意味的検索（セマンティック検索）
- ID検索（tabId での直接指定）

- **現状**: `get_sessions` 側は未実装。ただし `get_text_coords` には既に同じ三層（exact `search` / fuzzy `match` / MiniLM semantic）が実装済み（`read-helpers.js` の `fuzzyScore` + `embed-service`）。この既存基盤を `get_sessions` の URL/title に流用できる。

## 6. text-coords の取りこぼし対応（truncated + textarea value）  〔優先度: 高〕

text-coords の初期値は今のままで良い（軽量なサマリ、見えるテキスト優先）。
ただし agent側が **明示的に要求すれば** 取りこぼし分を取得できる手段を用意する：

1. 「[Pasted ~4 lines]」などで切り詰められた内容の全文取得
2. **textarea / input の `value`**（下記の理由で現状は範囲外）の取得

### textarea の中身が出ない理由（参考）

- text-coords は「画面上にレンダリングされたテキスト」（textContent / innerText ベース）を拾っている
- textarea の値は DOM テキストノードではなく `value` プロパティなので、text-coords の収集範囲外
- ブラウザ的には textarea の中身は「テキスト」として存在するが、text-coords が「見えるテキスト」にフォーカスしているから拾わない
- 拡張の実装次第で取ろうと思えば取れる（設計判断）

- **現状**: `text-coords.js`（shared 外、`extension/injected/` と `firefox-mv2/injected/` を両方編集が必要）は textarea を contextHint 分類（"text area"）には使うが、`value` は読んでいない。デフォルトは現状維持、`includeFormValues: true` のようなオプトインで `el.value` / `isContentEditable` を別フィールドに載せる方向。

## 7. 動的スキル（Dynamic Skills）  〔優先度: 低〕

各スキルについて、agentが詳しく知りたい場合により詳細な内容を見に行ける動的な仕組みを用意する。

- **現状**: 素地あり。`skills/browser-whiskor-http/` は `SKILL.md`（概要）＋ `reference.md`（詳細）の二段構成。「概要を読んで、必要なら reference を読みに行く」流れは既にこの形。拡張するなら、ツール/プロファイルごとの詳細ドキュメントへ agent が辿れる導線（MCP 側に `get_skill_detail` 的なもの、または reference のセクション分割）。やってもやらなくても良い、優先度低め。

## 8. 操作履歴の参照（低優先・今すぐ実装しなくて良い）  〔優先度: 低・保留〕

agent が過去の操作文脈を引けるようにする構想。

- 「このタブで直近5回の操作を見たい」（同一セッション内の操作履歴）
- 別タブでも **root が同じサイト** での操作履歴を横断参照
- サイトごとに1つの大きな log を持ち、条件付きで参照できるようにする
  - できる限りリアルタイム
  - 粒度をはっきり決める（通常の検索では細かすぎる粒度は含めない、等）

- **現状**: 記録基盤は既にある。`server/session-replay.js` が `actions.jsonl` にアクションを追記し、リプレイエンジンを持つ（`replay_session` ツール）。ただし agent 向けの「直近 N 件を読む」「サイト root 横断で集約して引く」参照 API は未実装。`state-store` / `state-navigator`（ステートグラフ）とも重なるので、設計時に役割分担を整理する。
- **方針**: 今は実装しない。構想メモとして保持。

## 9. OCR フォールバックによる視覚的ラベリング（aria/label 無し要素）  〔優先度: 中〕

text-coords は現状 **DOM ベース**（textContent/innerText + 座標、Tesseract 互換の座標フィールド形式）で、OCR エンジンは積んでいない。将来 OCR を**フォールバック**として足すなら、視覚にしか手掛かりのない要素（アイコンだけのボタン/リンク）に擬似ラベルを与えられる:

- aria も label も textContent も無い要素を `accessibility` / `clickability` で検出
- その要素のボックスを OCR にかけ、得られたテキストのうち **要素ボックス内に収まり、文字数に対してサイズがフィットする最も近いもの** を、そのボタンの **暫定ラベル** と仮定して紐付ける（断定せず confidence 付きの仮ラベル）
- agent はその暫定ラベルで操作してもよいが、それより **その要素だけをキャプチャして直接見る** ほうが確実（既存 `capture_element_screenshot` / `get_element_thumbnail`）
- 関連: 「SoM 発想A（未知UI自動ラベリング）」、項目6（text-coords 拡張）。OCR 導入は依存追加が重いので、まずは要素キャプチャ運用で足りるか見極めてから。

## 10. キャプチャ画像フォーマットの拡張（webp / AVIF）  〔優先度: 低〕

現状の capture 系（`capture_screenshot` / packed SoM / 要素サムネ）は `png` / `jpeg` のみ（`capture.js` の format enum）。

- **webp**: Chrome の canvas が `toDataURL('image/webp')` で対応。jpeg より小さく、トークン/転送量を削減できる。比較的容易
- **AVIF**: さらに小さいが canvas のエンコード対応はブラウザ差があり要検証（フォールバック設計が要る）
- 効きどころ: 画像を agent に返す全経路の payload 削減
