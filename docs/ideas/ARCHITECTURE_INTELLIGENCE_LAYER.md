  This document extends browser-whiskor v3 Architecture with the Intelligence
  Layer: a set of server-side and extension-side subsystems that transform raw
  collected data into deterministic conclusions before they reach the LLM agent.

  Design axiom: the agent receives conclusions, not raw material.  Every
  question the software can answer deterministically must be answered by the
  software.  Confidence values make the boundary explicit: a confidence of 1.0
  is a guarantee; anything below is a probability estimate with a stated basis.

  Dependency constraint: no npm packages beyond ws (server WebSocket) and
  bippy (bundled as lib/bippy.iife.js).  All algorithms are implemented in
  plain Node.js (server) and vanilla JS (extension injected world).

  SUBSYSTEM 1 : CSS ORIGIN TRACKER
??????????????????????????????????????????????????????????????????????????????

  Answers: "For a given DOM element, which CSS rule determines each computed
  property value, and from which source file does that rule originate?"

  Location: extension/injected/analyzers/css-origin.js
  Emit type: CSS_ORIGIN_MAP (new)
  Runs at: on-demand (triggered by agent tool call or manual collect)


  Source acquisition fallback chain
  ?????????????????????????????????

  The tracker operates at four acquisition levels.  Levels are attempted in
  order 1 ? 4.  The highest level successfully reached is recorded as
  acquisition_level in the output.

  ???????????????????????????????????????????????????????????????????????????
  ?  Level 1 ? chrome.devtools.inspectedWindow.getResources()               ?
  ?                                                                          ?
  ?  Precondition: DevTools panel is open and the extension has devtools    ?
  ?  page access.  Not available in content-script world; must be called    ?
  ?  from the DevTools extension context.                                   ?
  ?                                                                          ?
  ?  Capability: full source text of every resource (HTML, CSS, JS) as      ?
  ?  loaded, bypassing CORS.  Provides uncompressed source even for         ?
  ?  cross-origin stylesheets.                                              ?
  ?                                                                          ?
  ?  Confidence contribution: +0.00 (already at ceiling for CSS rules;      ?
  ?  source map resolution may push to 1.00 for original file/line).        ?
  ?  acquisition_level = 1                                                  ?
  ???????????????????????????????????????????????????????????????????????????

  ???????????????????????????????????????????????????????????????????????????
  ?  Level 2 ? document.styleSheets[i].cssRules traversal                   ?
  ?                                                                          ?
  ?  Available when: same-origin stylesheet, or cross-origin stylesheet     ?
  ?  served with Access-Control-Allow-Origin (enables CSSOM reflection).    ?
  ?                                                                          ?
  ?  Capability: full rule text via cssText, selectorText, style.*          ?
  ?  properties.  No original source line numbers.                          ?
  ?                                                                          ?
  ?  CORS blocked sheets: styleSheet.cssRules === null.  The tracker        ?
  ?  records { blocked: true, reason: "CORS" } and falls through to         ?
  ?  Level 3 for that sheet only.                                           ?
  ?                                                                          ?
  ?  acquisition_level = 2                                                  ?
  ???????????????????????????????????????????????????????????????????????????

  ???????????????????????????????????????????????????????????????????????????
  ?  Level 3 ? fetch(styleSheet.href)                                        ?
  ?                                                                          ?
  ?  Available when: the stylesheet URL responds with a permissive CORS     ?
  ?  policy or is same-origin.  The tracker inspects the                    ?
  ?  Access-Control-Allow-Origin response header before committing to a     ?
  ?  full fetch.                                                            ?
  ?                                                                          ?
  ?  The fetch is performed with { credentials: 'omit', cache: 'no-store' }.?
  ?  Failure modes: 4xx/5xx status, network error, CORS rejection.          ?
  ?  All failure modes fall through to Level 4.                             ?
  ?                                                                          ?
  ?  Capability: full source text.  No CSSOM reflection, so rule-to-line    ?
  ?  mapping requires text search on the fetched content.                   ?
  ?                                                                          ?
  ?  acquisition_level = 3                                                  ?
  ???????????????????????????????????????????????????????????????????????????

  ???????????????????????????????????????????????????????????????????????????
  ?  Level 4 ? URL and inline record only                                    ?
  ?                                                                          ?
  ?  Always available.  Records the stylesheet href (or "inline" for        ?
  ?  <style> elements) and the sheet index.  No rule text available.        ?
  ?                                                                          ?
  ?  acquisition_level = 4                                                  ?
  ???????????????????????????????????????????????????????????????????????????


  Rule matching algorithm
  ???????????????????????

  Given a target element el and a property name prop:

       1. window.getComputedStyle(el)[prop]           ? computed value V
       ?
       ?? Iterate document.styleSheets in reverse source order
       ?   (later sheets have higher precedence at equal specificity)
       ?
       ?? For each CSSStyleRule r in sheet.cssRules:
       ?     el.matches(r.selectorText) ?
       ?       YES ? r declares prop?
       ?               YES ? candidate: { rule: r, specificity: S(r.selectorText) }
       ?               NO  ? skip
       ?       NO  ? skip
       ?
       ?? Sort candidates descending by [specificity, source_order]
       ?
       ?? Winner = candidates[0]
       ?     Verify: winner.style[prop] === V (after normalization)
       ?     Mismatch indicates inherited value or browser UA sheet.
       ?
       ?? If no candidate matches: source is inherited or UA stylesheet.
             Record { source: "inherited" | "user-agent", confidence: 0.85 }

  Specificity computation (no external library):

    S(selectorText) ? [id_count, class_attr_pseudo_count, element_count]
    Computed by scanning the selector string with a single regex pass.
    Combined into a 32-bit integer: id*65536 + (class+attr+pseudo)*256 + element.
    Pseudo-elements (::before, ::after) count as one element.
    :not() content is counted; :not() itself is not.
    :is(), :where(): :is() counts the highest specificity of its arguments;
    :where() contributes zero specificity.

  Inline styles: always win regardless of specificity.  The tracker checks
  el.style[prop] !== "" before evaluating stylesheets.  An inline style
  produces { source: "inline", specificity: [1,0,0,0], confidence: 1.0 }.


  Confidence values for CSS Origin
  ?????????????????????????????????

    acquisition_level = 1 (getResources, source map resolved)   ? 1.00
    acquisition_level = 1 (getResources, no source map)         ? 0.97
    acquisition_level = 2 (cssRules, same-origin)               ? 0.93
    acquisition_level = 3 (fetch succeeded)                     ? 0.88
    acquisition_level = 4 (URL only)                            ? 0.40
    Inline style                                                ? 1.00
    Inherited / UA stylesheet                                   ? 0.85
    Multiple selectors match at equal specificity (tie)         ? ?0.08

  When multiple properties are returned in a single CSSStyleOriginMap, each
  property carries its own confidence value.  The map-level confidence field
  is the minimum across all property confidences.


  Output schema (CSSStyleOriginMap)
  ???????????????????????????????????

  Emitted as CSS_ORIGIN_MAP via the standard plugin emit path.

  {
    type: "CSS_ORIGIN_MAP",
    tabId: number,
    timestamp: number,
    element: {
      selector: string,          // computed unique selector for the element
      tag: string,
      id: string | null,
      classList: string[]
    },
    properties: {
      [propName: string]: {
        computedValue: string,
        source: "rule" | "inline" | "inherited" | "user-agent",
        rule: {
          selectorText: string,
          ruleText: string,             // full cssText of the matching rule
          specificity: number,          // packed 32-bit integer
          sheetHref: string | null,     // null for inline <style>
          sheetIndex: number,
          ruleIndex: number,
          sourceLine: number | null,    // null if not determinable
          originalFile: string | null,  // null if no source map
          originalLine: number | null
        } | null,
        acquisition_level: 1 | 2 | 3 | 4,
        confidence: number
      }
    },
    map_confidence: number          // min across all property confidences
  }

  SUBSYSTEM 2 : SOURCE LAYER
??????????????????????????????????????????????????????????????????????????????

  Answers: "What is the text content of a CSS or JS file referenced by the
  current page, and has it changed since the last session?"

  Images and binary assets are excluded.  Only text-bearing resources
  (MIME type text/css, text/javascript, application/javascript,
  application/x-javascript) are fetched and stored.

  Location (extension): extension/injected/analyzers/source-fetcher.js
  Location (server):    server/source-store.js
  Emit type:            SOURCE_CONTENT (new, replaces SOURCE_CATALOG for text files)


  Acquisition
  ???????????

  source-fetcher.js runs after SOURCE_CATALOG is populated.  It iterates
  catalog entries and attempts acquisition at the highest available level
  (mirrors CSS Origin Tracker levels 1?4).

  JS bundles: fetched and stored for hash comparison only by default.
  Full text storage for JS is opt-in (config.plugins.sourceFetcher.storeJs).
  Rationale: a typical SPA bundle is 500 KB?5 MB; storing it in the session
  cache for every session is wasteful.  The hash is sufficient for update
  detection.  Full text is available on explicit agent request via the
  get_source_file MCP tool.

  CSS files: fetched and stored in full.  Typical size 10?300 KB per file.
  Size cap: config.plugins.sourceFetcher.maxCssSizeBytes (default: 512 KB).
  Files exceeding the cap are stored as hash-only.


  Hash-based update detection
  ????????????????????????????

  On first acquisition for a URL, the tracker stores:

    { url, sha256, byteLength, acquiredAt, sessionId }

  in server/source-store.js (persisted to cache/sources/hashes.json).

  On subsequent sessions, the same URL is re-fetched (with If-None-Match
  using the stored ETag if available; otherwise unconditionally).  If the
  SHA-256 differs from the stored value, the tracker emits a SOURCE_CHANGED
  event:

    {
      type: "SOURCE_CHANGED",
      url: string,
      previousHash: string,
      currentHash: string,
      previousAcquiredAt: number,
      detectedAt: number,
      byteLength: { previous: number, current: number }
    }

  SOURCE_CHANGED is forwarded to the agent via the get_index response and
  optionally via a new MCP tool detect_site_updates.

  Invariant: the source ZIP from the reference extension
  (??(?)/??????????_??/) is not used at runtime.  That
  implementation was studied for its use of getResources(); the approach is
  adopted for Level 1 acquisition but the ZIP packaging is not replicated.
  source-fetcher.js emits individual SOURCE_CONTENT messages per file.


  Storage layout
  ??????????????

    cache/sessions/{siteVersion}/{tabId}-{sessionId}/
      raw/sources/
        catalog.json          ? existing: URL list (SOURCE_CATALOG)
        content/
          {sha256[0..7]}.css  ? CSS source text (if acquired)
          {sha256[0..7]}.js   ? JS source text (if storeJs = true)
    cache/sources/
      hashes.json             ? cross-session URL?hash registry

  SUBSYSTEM 3 : TIME-SERIES CORRELATOR
??????????????????????????????????????????????????????????????????????????????

  Answers: "Which network response caused which DOM mutation, and which
  framework state update mediated the relationship?"

  Location: server/correlator.js
  Input: existing timestamped events received by server/index.js
  Output: CausalChain objects stored in session cache, accessible via MCP


  Event bus
  ?????????

  The correlator maintains a per-tab ring buffer (CorrelationBuffer) of
  events.  Events are consumed from the existing WebSocket message stream in
  server/index.js without modification to the extension.

  ???????????????????????????????????????????????????????????????????????????
  ?  CorrelationBuffer                                                       ?
  ?                                                                          ?
  ?  Capacity:        200 events per tab                                    ?
  ?  Retention:       5 000 ms from event timestamp                         ?
  ?  Overflow policy: evict oldest event                                    ?
  ?  Thread model:    single-threaded (Node.js event loop); no locking       ?
  ?                   required                                              ?
  ???????????????????????????????????????????????????????????????????????????

  Event types consumed (all carry timestamp: number in their existing schema):

    NETWORK_RESPONSE     ? { requestId, url, status, responseKeys[], timestamp }
    REACT_SNAPSHOT       ? { rootComponents[], timestamp }
    VUE_SNAPSHOT         ? { components[], timestamp }
    DOM_GENERIC_SNAPSHOT ? { mutationCount, timestamp }  (proxy; see note)
    TEXT_COORD_DELTA     ? { deltas[], timestamp }
    EXPLORER_STATE_UPDATE? { compositeHash, timestamp }

  Note on DOM mutation source: the extension emits TEXT_COORD_DELTA and
  DOM_GENERIC_SNAPSHOT rather than MutationObserver events directly.  The
  correlator uses TEXT_COORD_DELTA as its DOM-mutation proxy because it
  carries per-element coordinate deltas with timestamps, making temporal
  correlation tractable.  A dedicated DOM_MUTATION event type may be added
  in a later phase to improve precision.


  Correlation rules
  ?????????????????

  Rules are evaluated in the order listed.  Each rule produces a
  CausalChain if its conditions are met.  A single DOM mutation event may
  match multiple rules; all matching chains are recorded.

  Rule 1 ? Network ? DOM
  ???????????????????????????????????????????????????????????????????????????
  ?  Trigger:   NETWORK_RESPONSE followed by TEXT_COORD_DELTA               ?
  ?  Window:    deltaMs ? 500                                               ?
  ?  Confidence base: 0.70                                                  ?
  ?  Adjustments:                                                           ?
  ?    deltaMs ? 50  ms          ? +0.15                                    ?
  ?    deltaMs ? 150 ms          ? +0.08                                    ?
  ?    responseKeys ? affectedTextTokens ? ?  ? +0.10                      ?
  ?    status ? 200             ? ?0.10  (error response is weak cause)    ?
  ?  Minimum reported confidence: 0.50 (chains below this are discarded)   ?
  ???????????????????????????????????????????????????????????????????????????

  Rule 2 ? Framework update ? DOM
  ???????????????????????????????????????????????????????????????????????????
  ?  Trigger:   REACT_SNAPSHOT or VUE_SNAPSHOT followed by TEXT_COORD_DELTA ?
  ?  Window:    deltaMs ? 100                                               ?
  ?  Confidence base: 0.85                                                  ?
  ?  Adjustments:                                                           ?
  ?    Framework?DOM linkage confirmed by Framework?DOM Mapper ? 1.00      ?
  ?    (exact match: no probabilistic inference needed)                     ?
  ?    deltaMs ? 20  ms          ? +0.08                                    ?
  ?    Multiple components updated ? ?0.05 (ambiguous actor)               ?
  ???????????????????????????????????????????????????????????????????????????

  Rule 3 ? Network ? Framework ? DOM (composed chain)
  ???????????????????????????????????????????????????????????????????????????
  ?  Trigger:   Rule 1 chain AND Rule 2 chain where Rule 2 framework        ?
  ?             event falls within the Rule 1 window                        ?
  ?  Confidence: min(Rule1.confidence, Rule2.confidence) + 0.05             ?
  ?  This is the most complete causal chain the system can produce.         ?
  ???????????????????????????????????????????????????????????????????????????


  False-positive mitigation
  ??????????????????????????

  Time-based correlation is inherently probabilistic.  The following
  heuristics reduce false positives:

    Background polling:  requests with the same URL recurring at a regular
      interval (? 3 occurrences, coefficient of variation of inter-arrival
      time < 0.15) are tagged { pollingCandidate: true } and their
      correlation confidence is halved.  Polling responses are unlikely to
      be the meaningful cause of a specific DOM change.

    Unrelated mutations:  TEXT_COORD_DELTA events that correspond only to
      scroll-classified motion groups (from the Delta Engine) are excluded
      from correlation.  Scroll is driven by user input, not network events.

    Confidence floor:  chains with final confidence < 0.50 are discarded and
      never written to cache.  The agent never receives sub-threshold chains.


  CausalChain schema
  ???????????????????

  {
    id: string,                    // sha256(events joined)[0..8]
    rule: 1 | 2 | 3,
    confidence: number,            // 0.50 ? 1.00
    network: {                     // present for rules 1 and 3
      requestId: string,
      url: string,
      method: string,
      status: number,
      deltaMs: number              // ms from network response to DOM mutation
    } | null,
    framework: {                   // present for rules 2 and 3
      type: "react" | "vue" | "angular" | "svelte" | "solid" | "other",
      components: string[],        // component names that updated
      deltaMs: number
    } | null,
    dom: {
      affectedCount: number,       // number of elements in TEXT_COORD_DELTA
      sampleSelectors: string[],   // up to 3 computed selectors of affected els
      deltaMs: number              // ms from trigger event to delta
    },
    timestamp: number,
    sessionId: string
  }

  Stored at: cache/sessions/.../raw/intelligence/causal-chains.json
  Max stored: 500 chains per session (LRU eviction on overflow).

  SUBSYSTEM 5 : CLICKABILITY ANALYZER
??????????????????????????????????????????????????????????????????????????????

  Motivation: when an LLM agent attempts to click an element that is blocked
  by a modal dialog (for example, an MUI Dialog overlaying a text input),
  the current executor returns only { ok: false, error: "Element not found" }
  or a generic error.  The agent has no basis for diagnosing the cause and
  enters a retry loop.  The Clickability Analyzer produces a structured
  report before and after each click attempt, giving the agent a
  deterministic account of why a click failed and what must happen first.

  Location: extension/injected/analyzers/clickability.js
  Invoked by: extension/injected/executor.js (pre- and post-click)

  Pre-click analysis
  ??????????????????

  analyzeClickability(el) ? ClickabilityReport

  All checks use synchronous DOM/CSSOM APIs.  No async operations.

  Check 1 ? Existence
  ???????????????????????????????????????????????????????????????????????????
  ?  el !== null && document.contains(el)                                   ?
  ?  Failure: { exists: false }. Remaining checks skipped.                  ?
  ???????????????????????????????????????????????????????????????????????????

  Check 2 ? Visibility
  ???????????????????????????????????????????????????????????????????????????
  ?  cs = window.getComputedStyle(el)                                       ?
  ?  visible = cs.display !== 'none'                                        ?
  ?         && cs.visibility !== 'hidden'                                   ?
  ?         && cs.visibility !== 'collapse'                                 ?
  ?         && parseFloat(cs.opacity) > 0                                   ?
  ?                                                                          ?
  ?  Also walks ancestors: a hidden ancestor makes the element invisible    ?
  ?  regardless of its own computed style.                                  ?
  ?  Ancestor walk limit: 32 levels (prevents pathological DOM depth).      ?
  ?                                                                          ?
  ?  Failure: { visible: false, hiddenBy: { selector, property, value } }  ?
  ???????????????????????????????????????????????????????????????????????????

  Check 3 ? Viewport presence
  ???????????????????????????????????????????????????????????????????????????
  ?  rect = el.getBoundingClientRect()                                      ?
  ?  inViewport = rect.width > 0 && rect.height > 0                         ?
  ?            && rect.bottom > 0 && rect.right > 0                         ?
  ?            && rect.top < window.innerHeight                             ?
  ?            && rect.left < window.innerWidth                             ?
  ?                                                                          ?
  ?  Failure: { inViewport: false, rect }.  The executor will attempt       ?
  ?  scrollIntoView before proceeding.                                      ?
  ???????????????????????????????????????????????????????????????????????????

  Check 4 ? Pointer events
  ???????????????????????????????????????????????????????????????????????????
  ?  cs.pointerEvents === 'none'                                            ?
  ?  Applies to the element and its ancestors (same walk as Check 2).       ?
  ?  pointer-events: none on an ancestor blocks events on all descendants. ?
  ?                                                                          ?
  ?  Failure: { pointerEventsEnabled: false, blockedBy: { selector } }     ?
  ?  Strategy implication: 'programmatic' click required.                   ?
  ???????????????????????????????????????????????????????????????????????????

  Check 5 ? Disabled state
  ???????????????????????????????????????????????????????????????????????????
  ?  disabled = el.disabled === true                                        ?
  ?          || el.getAttribute('aria-disabled') === 'true'                ?
  ?          || el.getAttribute('disabled') !== null                        ?
  ?                                                                          ?
  ?  Failure: { disabled: true }.  No click strategy will produce a         ?
  ?  meaningful result; the analyzer reports canAutoFix: false.             ?
  ???????????????????????????????????????????????????????????????????????????

  Check 6 ? Obstruction (hit-test)
  ???????????????????????????????????????????????????????????????????????????
  ?  rect  = el.getBoundingClientRect()                                     ?
  ?  cx    = rect.left + rect.width  / 2                                   ?
  ?  cy    = rect.top  + rect.height / 2                                   ?
  ?  topEl = document.elementFromPoint(cx, cy)                              ?
  ?                                                                          ?
  ?  obstructed = topEl !== null                                            ?
  ?            && !el.contains(topEl)                                       ?
  ?            && el !== topEl                                              ?
  ?                                                                          ?
  ?  When obstructed:                                                       ?
  ?    Walk topEl ancestors for [role="dialog"], [role="alertdialog"],      ?
  ?    elements matching .modal, .overlay, .backdrop, [data-modal],         ?
  ?    MUI-specific: .MuiDialog-root, .MuiBackdrop-root, .MuiPopover-root  ?
  ?    First match ? obstructorInfo.isModal = true                          ?
  ?                                                                          ?
  ?  Failure: { obstructed: true, obstructedBy: ObstructorInfo }           ?
  ?                                                                          ?
  ?  Note: elementFromPoint uses the center of the bounding rect.           ?
  ?  For non-rectangular elements (CSS clip-path, border-radius) the center ?
  ?  may not represent the true hit region.  This is a known limitation.    ?
  ???????????????????????????????????????????????????????????????????????????

  ?????????????????

  ????????????????????????????????????????????????????????????????????????
  ?  Strategy        ?  Mechanism                                        ?
  ????????????????????????????????????????????????????????????????????????
  ?  native          ?  Dispatch MouseEvent sequence:                    ?
  ?  (default)       ?  pointerover ? mouseover ? pointermove ?         ?
  ?                  ?  mousemove ? pointerdown ? mousedown ?           ?
  ?                  ?  pointerup ? mouseup ? click                     ?

  ?                  ?  and clientX/Y from getBoundingClientRect center ?
  ????????????????????????????????????????????????????????????????????????

  ?                  ?  Bypasses pointer-events: none.                   ?

  ?                  ?  fiber.memoizedProps.onClick?.({ ... })           ?
  ?                  ?  Vue: instance.vnode.props?.onClick?.()           ?
  ?                  ?  Fires the framework handler directly.            ?
  ?                  ?  Browser default actions (form submit, link nav) ?
  ?                  ?  are NOT triggered.  Record in diagnosis.         ?
  ????????????????????????????????????????????????????????????????????????
  ?  force           ?  1. Hide each obstructor: el.style.display='none'?
  ?                  ?  2. Execute 'native' click on target              ?
  ?                  ?  3. Restore: el.style.display=''                 ?
  ?                  ?  Risk: may leave modal in inconsistent state.     ?
  ?                  ?  Used only when canAutoFix=true AND all unblock   ?
  ?                  ?  attempts have failed.                            ?

    ??????????????????????????????????????    ????????????????????

  Auto-unblock pipeline
  ??????????????????????

  Executed only when obstructed=true AND isModal=true.
  Attempted in order; stops at first success.

       Step 1: Click closeButtonSelector (if hasCloseButton=true)
       ?         Confirm: obstructor no longer in DOM or no longer visible
       ?
       Step 2: Dispatch KeyboardEvent 'Escape' on document
       ?         Confirm: obstructor removed/hidden within 300 ms
       ?
       Step 3: Click the backdrop element
       ?         Selector: .MuiBackdrop-root, [data-overlay], .modal-backdrop,
       ?                   the computed backdrop sibling of the dialog
       ?         Confirm: as Step 2
       ?
       Step 4: Set canAutoFix = false, fixResult = "all_steps_failed"
               Return report without executing target click.

  Each step confirmation uses a 300 ms MutationObserver wait scoped to the
  obstructor element.  If the obstructor is removed from the DOM or its
  computed display becomes 'none', the step is considered successful.

  KNOWN LIMITATIONS AND OPEN QUESTIONS
??????????????????????????????????????????????????????????????????????????????

  CSS Origin ? elementFromPoint center heuristic
    The rule-matching algorithm uses el.getBoundingClientRect() center for
    elementFromPoint in Check 6 of the Clickability Analyzer.  The CSS Origin
    Tracker itself does not use elementFromPoint; it matches rules via
    el.matches().  Non-rectangular elements (clip-path, custom shapes) may
    cause elementFromPoint to miss the element in Click Check 6 only.

  CSS Origin ? @layer and @scope (CSS Cascade Layers, CSS Scoping)
    Cascade Layers (CSS @layer) alter the cascade order independently of
    specificity.  The current specificity computation does not account for
    @layer ordering.  Rules inside a later @layer override rules in earlier
    @layer at equal specificity.  This is a known gap; the tracker will
    report the correct computed value from getComputedStyle but the
    winning-rule identification may be incorrect for layered stylesheets.
    Resolution: post-process candidates against CSSLayerStatementRule order.

  Time-series Correlator ? DOM mutation proxy precision
    TEXT_COORD_DELTA is used as a DOM mutation proxy.  It is generated by
    the Delta Engine from visual coordinate changes, not from MutationObserver
    directly.  Mutations that produce no coordinate changes (attribute-only
    mutations, invisible insertions) are not observed.  A future
    DOM_MUTATION event type would improve coverage.

  Framework?DOM Mapper ? production Svelte
    Svelte production builds emit no __svelte_meta annotations.  The mapper
    falls back to acquisition level 5 (DOM position only) for production
    Svelte applications.  There is no known workaround without modifying the
    Svelte compilation output.

  Correlator ? async renders
    Frameworks with async rendering (React concurrent mode, Vue async
    components) may produce DOM mutations hundreds of milliseconds after the
    triggering network response.  The 500 ms window (Rule 1) is intentionally
    wide to accommodate this.  Increasing the window reduces false negatives
    at the cost of increased false positive rate.

  Clickability ? programmatic strategy and browser defaults
    The programmatic click strategy invokes the framework event handler
    directly.  Browser-native behaviors tied to the full event chain
    (form validation on submit, link navigation, drag-and-drop default
    actions) are not triggered.  The diagnosis field records
    unexpectedBehavior: "default_prevented" when this strategy is used.

  Source Layer ? JS bundle size
    Full-text storage of JS bundles is disabled by default (storeJs=false).
    Typical SPA bundles range from 200 KB to 5 MB per chunk; storing all
    chunks would dominate the session cache.  Agents that need JS source
    content should use get_source_file on-demand.
