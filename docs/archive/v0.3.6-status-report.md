## Quick summary

- ✅ **State-hash stability fixed.** The live React state hash now applies a
  **key-aware non-deterministic filter** (default), so volatile props
  (timestamps, nonces, UUIDs) no longer churn the hash while legitimate numeric
  IDs survive. Configurable via `config.json` → `react.hashFilter.mode`
  (`off` / `key-aware` / `aggressive`).
- ✅ **`observe` made reliable.** Post-action settle uses **adaptive polling +
  a quiescent window** instead of a fixed 200 ms loop, so fast SPA transitions
  aren't missed and a quick A→B→A flip doesn't settle early. Configurable via
  `config.json` → `observe` (`adaptive: false` restores legacy behaviour).
- ✅ **Docs refreshed.** README updated to v0.3.6; `architecture.md` /
  `agent-knowledge.md` ND-filter sections updated to the key-aware model.

### Notable finding

The React state hash was computed in **two places**: the live client path
(`react.js`, djb2, **no ND filter**) and a dead server path
(`state-fingerprint.js`, FNV-1a, with the documented 13-digit heuristic). Only
the unfiltered client path actually fed `observe`/explorer, which is why the
hash was unstable. v0.3.6 fixes the live path and aligns the server spec. See
[`v0.3.6-improvements.md`](v0.3.6-improvements.md) §1 for details.
