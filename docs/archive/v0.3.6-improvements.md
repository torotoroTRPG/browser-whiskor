# Browser-Whiskor v0.3.6 Improvements

## Overview

This document records the two changes shipped in v0.3.6 **and the reasoning
behind them** — in particular why the post-action observation uses adaptive
polling instead of a WebSocket "push" model, and why the state-hash filter is
key-aware rather than a blind numeric heuristic.

Both changes share a single root cause: **the state hash that `observe` and the
explorer actually consume was unstable**, because the live hash had no
non-deterministic (ND) value filtering at all.

---

## 1. State-hash stability — key-aware ND filter

### Problem

The React state hash is computed in **two different places**, and only the dead
one had an ND filter:

| | Live path (consumed by observe/explorer) | Server path |
|---|---|---|
| File | `shared/injected/adapters/react.js` `_getStateHash` | `server/state-fingerprint.js` `computeReactHash` / `filterNd` |
| Algorithm | djb2 (`_hash`) | FNV-1a |
| ND filter | **none** — every primitive prop fed into the hash | 13-digit / UUID / 32+ random heuristic |
| Status | writes `window.__SI_REACT_HASH__` → explorer `computeCompositeHash` → `observe` | **dead code** — only `getStoreKeys` is called anywhere |

Consequences:

1. The documented "non-deterministic filter (13-digit timestamps, UUIDs…)" only
   existed in `state-fingerprint.js`, which **nothing calls**. The live path did
   no filtering, so any volatile prop (a timestamp, a counter, a nonce) changed
   the hash on every React commit. This produced phantom state transitions and
   was the direct cause of `observe` never settling (§2).
2. The two implementations use different hash algorithms, so they can never
   agree — the server-side functions could not have validated the client hash
   even if they were wired up.
3. The blind `13-digit number → strip` heuristic in the dead code was itself
   fragile: a legitimate 13-digit numeric id (order id, snowflake id) would be
   silently discarded, merging genuinely distinct states.

### Solution

Implement ND filtering **in the live path** (`react.js`) and mirror the same
spec server-side, controlled by `config.json` → `react.hashFilter.mode`.

The filter is **key-aware** by default: a value is only treated as volatile when
its *key name* looks volatile, or the value matches an unambiguous format.

### Implementation

**Files:**
- `shared/injected/adapters/react.js` — `_ndContext` / `_ndNormalize` /
  `_ndIsTemporalKey`, applied in `_getStateHash` + `_getTreeShape` (the live path)
- `server/state-fingerprint.js` — `filterNd` refactored to the same key-aware
  spec (+ ISO-8601, key-context propagation)
- `config.json` — new `react.hashFilter` block (reaches the page via
  `server/index.js` spreading `react` into `options.react`)

**Modes:**

| Mode | Behaviour |
|------|-----------|
| `off` | No filtering (legacy live behaviour; hash changes on every volatile prop). |
| `key-aware` *(default)* | Strip a value only when its key is temporal (`createdAt`, `*At`, `timestamp`, `nonce`, `_ts`…) **or** the value is an unambiguous UUID v4 / ISO-8601 datetime. **Legitimate numeric IDs — even 13-digit — survive.** Configurable `excludeKeys` are dropped entirely. |
| `aggressive` | Additionally strip bare 13-digit numbers and 32+ char random strings regardless of key (restores the old blind heuristic for maximum stability). |

### Decision rationale

- **Why fix the client, not the dead server code?** The hash that observe and
  the explorer read is produced client-side and stored opaquely by the server
  (`state-store.js` saves the reported string verbatim). Patching
  `state-fingerprint.js` alone would have changed *nothing* observable. We still
  aligned the server filter to the same spec so the documented behaviour is true
  and a future server-side recompute would match.
- **Why key-aware over blind 13-digit?** The user's concern was real: a blind
  `\d{13}` rule discards valid IDs. Keying the rule to the *field name* keeps
  timestamps out of the hash while preserving IDs, which is what actually
  distinguishes states. UUID/ISO are stripped unconditionally because they are
  unambiguous and were already the documented behaviour.

### Compatibility note

Switching the default to `key-aware` changes hash *values* versus the previous
(unfiltered) live behaviour, so existing state-graph nodes re-bucket. This is
benign — the graph simply re-learns with more stable hashes. Set
`mode: "off"` for the exact legacy hash, or `"aggressive"` for the old
state-fingerprint heuristic.

## 2. Post-action observation — adaptive settle loop

### Problem

`observe` watches the page state hash after an action and reports whether the UI
transitioned. The original loop polled at a **fixed 200 ms** and declared the
state "settled" after two consecutive equal reads. Two failure modes:

1. **Missed fast transitions** — a React re-render completes in a few ms, but the
   first poll only happens 200 ms later, so a quick `A → B → A` could be missed
   or mis-snapshotted.
2. **Never settling** — because the live hash had no ND filter (§1), volatile
   props kept changing it, so two equal reads in a row rarely happened.

### Solution

Make the settle loop **adaptive** and add a **quiescent window**, both
config-driven. Combined with §1 (a now-stable hash) this is reliable.

### Implementation

**File:** `server/mcp/tools/write.js` — `_awaitSettled` / `_observeOpts`,
config via `config.json` → `observe`.

- **Adaptive intervals** `[60, 60, 120, 200]` ms — poll fast at first to catch
  brief transitions, then back off to keep RPC cost low.
- **Quiescent window** `quiescentMs: 150` — require `settleReads` consecutive
  equal reads **and** at least 150 ms of quiet since the last change, so a fast
  `A → B → A` flip resets the window instead of settling early.
- **Legacy mode** `adaptive: false` — fixed `intervalMs` with immediate settle on
  N equal reads, identical to the original behaviour. The quiescent window is
  forced to 0 in this mode.
- The observation result gained `reads` and `mode` fields (additive).

### Decision: WebSocket "push" was considered and rejected

The alternative to polling was an **event-driven push**: tell the page once to
start watching, and have it push a `STATE_HASH_REPORT` over WS `:7891` whenever
the hash changes, instead of the server asking repeatedly.

```
poll (chosen):  server ──REQUEST_STATE_HASH──▶ page ──STATE_HASH_REPORT──▶ server   (repeat)
push (rejected): server ──REQUEST_STATE_HASH {watchMode:true}──▶ page
                 page ──(on change)──▶ STATE_HASH_REPORT {requestId:'watch'}──▶ server
                 server ──CANCEL_WATCH──▶ page
```

**The plumbing is already half-built** in `shared/injected/state-reporter.js`
(`startWatchMode` / `CANCEL_WATCH`), so it is tempting. It was rejected because:

1. **No server-side consumer.** `state-navigator.js:requestHash` resolves pending
   requests by `requestId`; the `requestId: 'watch'` pushes match nothing and are
   dropped. Wiring it up requires a per-tab subscription registry plus lifecycle
   management (start / cancel / timeout / cleanup-on-unload) threaded through
   MAIN world → bridge → SW → WS → core routing → navigator. That is a
   cross-layer change with real risk.
2. **It is not actually event-driven.** `startWatchMode` polls
   `window.__SI_CURRENT_HASH__` on its own `setInterval(…, 200)`. The "push"
   only moves the polling to the client and adds change-dedup — it is not hooked
   to React's commit lifecycle, so it has no latency advantage over §2.
3. **§1 makes polling sufficient.** Once the hash is stable, adaptive server-side
   polling reaches the same conclusion with far less moving machinery.

**If true push is ever wanted**, the correct signal source is *not*
`state-reporter.js`'s timer but `react.js`'s `onCommitFiberRoot` (which already
emits `REACT_TRANSITION` on the exact commit where the hash changes). That is the
only place that knows the instant a state truly changed.

## Configuration reference

```jsonc
{
  "react": {
    "hashFilter": {
      "mode": "key-aware",   // "off" | "key-aware" | "aggressive"
      "excludeKeys": []       // extra prop names to drop entirely
    }
  },
  "observe": {
    "adaptive":    true,           // false → legacy fixed-interval behaviour
    "intervalsMs": [60, 60, 120, 200],
    "intervalMs":  200,            // used only when adaptive=false
    "settleReads": 2,
    "quiescentMs": 150             // min quiet after last change (adaptive only)
  }
}
```
