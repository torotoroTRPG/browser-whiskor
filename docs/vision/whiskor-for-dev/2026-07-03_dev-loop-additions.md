## 前提（原点確認）

whiskor はもともと「自分の開発を楽にするため、かつ汎用的ならなお良い」で始まった面がある。
つまり whiskor-for-dev は方向転換ではなく**出自への回帰**。以下の項目はその線での追加口述。

## 1. ローカルバックエンドとの対話 — 「観測」に加えて「代役」の軸

### 口述の趣旨
専用ブラウザ（engine-embed）構想の本質の一つは「ページと、**ブラウザ外にある開発者想定のローカルバックエンド**（Tauri の Rust プロセス等）との間で whiskor がやりとりできること」。

### 既存構想との関係
- LAYER 4 ＋ reporter（[`full-stack-trace.md`](full-stack-trace.md) 予定地）は「実バックエンドの span を**観測**する」軸。✅ 既出。
- **新規の軸:** whiskor server が実バックエンドの**代役（stand-in）／中継（relay）**になること。
  - **relay モード:** ページ ⇔ 実バックエンド間の WS/HTTP/IPC を中継し、契約（メッセージスキーマ・順序）を記録・アサート。
  - **emulate モード:** 実バックエンド不在でも、記録済み契約から応答を再生。**ビルド前検証**（フロント側だけ差し替えて動作確認）が成立する。

### 具体的活用例
- **Tauri 等の wrapper アプリ開発:** injected script は拡張ホストで素のブラウザに載せ、Rust バックエンドは whiskor が emulate。Rust をビルドせずにフロント側機能の検証ループが回る。
- 通常の web dev でも「モックサーバー＋通信ログ＋契約アサート」を一体で持てる。

### 区別と制約（重要）
- **relay は reporter（D9）の代替ではない。** relay が見えるのは**ワイヤ上の通信（WS/HTTP）だけ**で、ハンドラ内部の span・バックエンド内部状態は見えない。さらに Tauri の `invoke()` IPC はネットワークを通らないため **relay では原理的に見えない** — これは D9 が指摘するギャップそのもの。relay ＝ 導入コスト 0 の浅い可視性、reporter ＝ 統合コストありの深い可視性。**補完関係であり、relay の存在を D9 後退の根拠にしないこと。**
- **emulate の契約記録は認証情報を吸い込む。** ワイヤ通信をそのまま記録・永続化すると、トークン・cookie・API キーがディスクに落ちる。これは D8 が「復号移植しない」とまで慎重に守っている境界と、`secret-guard.js` が守っている redaction 境界の**両方を横から迂回する**。設計制約:
  1. 契約記録は必ず secret-guard の redaction を通す（秘密は名前付きプレースホルダ＝ secret slot として記録し、replay 時に差し替え）
  2. **認証交換そのものは emulate の対象外**。「認証済み」は状態として `assume-filled` 宣言する（＝より上のレイヤーでスタブし、認証フローの再生はしない）

### 現状の足場
- server は元々ブラウザ外の常駐 Node（ws:7891 / http:7892）。中継点としての位置は既にある。
- `mock-data.js`（--mock モード）が emulate の原始形。
- `secret-guard.js`（server 側 redaction）＝上記制約 1 の通過点として既にある。
- `action-executor.js` / `delta-engine.js` の配管はそのまま使える。

## 2. リアルタイム編集 → 「動くか」の判定ループ

### 口述の趣旨
web app に編集を加えたとき「動くか・実際に見た目がどうなるか」をリアルタイムに判定したい。

### 既存構想との関係
- loop closure（[`loop-closure.md`](loop-closure.md) 予定地）は **agent の操作**に expectation を付けて採点する軸。✅ 既出。
- **新規の軸:** トリガーが操作ではなく**ファイル編集（HMR）**である変種。
  - baseline snapshot → 編集検知（HMR イベント or FS watch）→ 安定待ち → 差分収集 → 「変更は効いた／何も壊れてない」の構造化 verdict。
  - expectation なしでも「console エラー新規発生ゼロ・状態グラフの意図しない遷移ゼロ」という**デフォルト判定**が出せる。

### 現状の足場
- `delta-engine.js`（スマート差分集約・motion clustering）
- `state-fingerprint.js`（FNV32 状態ハッシュ）＝「同じ画面に戻ったか」の判定に流用可
- `correlator.js`（時系列因果チェーン）＝「この編集がこの差分を生んだ」の紐付け
- 欠けているのは **編集イベントをトリガーとして受ける口**（HMR フック or FS watcher）と verdict のレポート形式だけ。組み合わせ問題に近い。

## 3. ヘッドレス ＋ 論理キャプチャを「主表示」にする

### 口述の趣旨
基本機能は画面アリ前提でよいが、ヘッドレスでも**論理キャプチャで画面を見せる**運用が成り立つはず。

### 既存構想との関係
- discussion **D12**（headless＝自律 verify/CI 向き）で headless 自体は既出。
- **追記:** headless 時の一級の「画面」をピクセルではなく**論理表現**にする、という表示側の方針。
  - layout-map（ASCII 空間マップ）＋ UI カタログ ＋ a11y ツリー ＋ text-coords を束ねた「論理スクリーン」を capture 系の一形態として返す。
  - ピクセル screenshot が取れない/要らない文脈（CI、リモート、トークン節約）での既定表示。

### 現状の足場
- `layout-map.js` / UI catalog / accessibility / text-coords は全て実装済み。個別ツールとしては既にある。
- 欠けているのは「headless プロファイルではこれらを合成して一発で返す」**パッケージング**。
- 注意: 論理キャプチャは「見た目の最終確認」の代替にはならない（色・重なり・フォント崩れは落ちる）。役割は**構造の確認**。ピクセル確認は画面アリ or CI スクショに残す。

## 4. focus 駆動のモデル分析・保持 — IntelliJ 的「変数の流れ」をランタイム実測で

### 口述の趣旨
純粋に collect した内容に対してモデルを通し、IDE（IntelliJ 等）の解析のように**分析結果を保持**する。
たとえば人間/AI がどこかに**注目した瞬間**、その変数の流れを**論理コンポーネント単位で丸ごと**実用的に確認・追跡・操作できる、というもの。

### 既存構想との関係
- LAYER 1 の「def-use を観測」、context-brain（correlator / conclusion-cache / source-correlation、中核は v3 実装済み）、dynamic-focus（視覚的な注視）に**それぞれ半分ずつ**既出。
- **新規なのは束ね方:** 「focus をトリガーに、1 論理コンポーネントの dossier（一件書類）を組み立ててモデルに通し、結果を保持する」という一手。
  1. focus 対象（コンポーネント/要素/状態キー）を受ける
  2. 収集: fiber サブツリー ＋ hooks/state の時系列 ＋ dispatch されたアクション ＋ 関連 network ＋ source-correlation のソース片
  3. モデル分析: 「この値はどこで生まれ、どこを通り、どこで描画に効くか」を def-use 物語として要約
  4. 保持: conclusion-cache に **component × state-fingerprint × source hash** をキーに格納。同じ状態で再訪したら再分析せず返す。**source hash は必須** — dev 文脈では編集が常態（項目 2 と併用される前提）なので、コード変更後に古い分析を返したら実害が出る。ソース/成果物ハッシュで無効化する
- 静的 IDE のデータフロー解析との差別化: こちらは**実際に走った値**で裏取りされる（LSP が原理的に持てない情報）。README のテーゼ「runtime 側半分」の最も直接的な体現。

### 現状の足場
- `source-correlation.js` / `source-index.js` / `source-store.js`（runtime↔source 対応付け）
- `conclusion-cache.js`（保持層そのもの）
- intelligence tools（explain_element / why_did_this_change）＝ dossier の部品収集の前例
- dynamic-focus 構想 ＝ focus という入力語彙の前例（あちらは視覚、こちらは意味。統合できると良い）

## 5. 開発者アダプタ — 「配線」そのものを製品機能にする

### 口述の趣旨（2026-07-03 追記）
whiskor を開発者ツールとして使うなら、プロジェクトごとに手作業で配線するのではなく、**汎用かつ網羅的な開発者用アダプタ**を専用に作った方がよいのでは。
- 専用 config で**大雑把にも細かくにも**調整できること（セキュリティ案件なので粒度制御は必須）
- 開発中の連携の例: 「うちは TS」→ ソースを投げ込めばアダプタ側でビルドして注入まで通る（**TS は一例**。言語・toolchain には依存させない — ビルド手順はプロジェクト宣言側が持ち、アダプタは「宣言されたコマンドを起動し成果物を受け取る」だけの立場。wasm でも別トランスパイラでも同じ口に乗る）
- **足りない部分の扱い:** 「足りない部分はどうするの」→ **足りる前提で無視できる**（未実装の依存を宣言しておけば、そこはスタブされて残りの開発ループが今日から回る）

### 整理: これは「統合契約（integration contract）」
項目 1（backend 代役）と 2（編集→判定）は、実はこのアダプタの**部分機能**。アダプタは両者の入口になる。

構成要素の想定:
- **プロジェクト宣言**（プロジェクト側に置く manifest、例 `whiskor.dev.json`）
  - toolchain: ビルド/watch コマンド、成果物と注入先のマッピング（例: TS → bundle → どのページに inject するか。言語非依存で、宣言されたコマンドを叩くだけ）
  - dependencies: 各依存の状態を宣言 — `present`（実物あり）/ `relay`（中継・記録）/ `emulate`(記録から再生) / `assume-filled`（未実装。スタブして無視 = 「足りる前提」）
  - capabilities: whiskor に許す操作の範囲
- **whiskor 側アダプタランタイム**: 宣言を読んで注入・スタブ生成・判定ループ（項目2）を配線する
- **セキュリティ設計**: default-deny。粗い粒度は preset（`observe-only` / `dev-full` / `ci`）、細かい粒度は per-capability の個別 on/off。localhost 外のターゲットには dev 系 capability を出さない

### スコープの締め（non-goal を先に切る）
- **アダプタはビルドオーケストレータではない。** ビルドの所有・依存グラフ・並列化・キャッシュは既存ツール（npm scripts / turbo / 任意）の領分で、そこを薄く再発明しない。アダプタが所有するのは**「成果物とランタイム観測ループの結線」だけ** — inject mapping・依存スタブ・capability gate・verdict 配線。
- 立場としては「宣言されたコマンドを起動できる」は便宜オプションに格下げし、本体は **"artifact ready" イベントの消費者**（watch/HMR が吐いた成果物を受けて注入する側）。
- 固有の強み＝汎用ビルドツールが構造的に持たないもの: **成果物を「本物のページランタイム」に届けて、その後の観測・判定まで一続きで面倒を見る**こと。ここだけが whiskor の土俵で、ここから外れる要求は既存ツールに任せる。

### 開発者モード＝明示的・可視・一時
dev 系 capability は常設 config ではなく**モード**として扱う:
- **明示的に入る**（コマンド/UI 操作で開始。プロジェクト単位・セッション単位で scoped）
- **常時可視**（拡張アイコンの色反転・バッジ等で「今このブラウザは dev モード」が一目で分かる。録画インジケータと同じ発想）
- **一時的**（セッション終了 or タイムアウトで自動失効。`config-change-log.js` の監査＋auto-revert の前例にそのまま乗る）
- 対象 origin は localhost/宣言済みプロジェクトに限定

### 現状の足場
- `server/configs/mcp-tools.json` ＋ `config-loader.js`: per-tool 可視性・preset の config パターンがそのまま前例になる
- `app-registry.js`（appIsolation）: プロジェクト単位の分離の前例
- `config-change-log.js`: config 監査・自動 revert — セキュリティ要件側の部品も既にある
- 欠けているのは「プロジェクト側 manifest の語彙定義」と「toolchain hook（外部ビルドの起動・成果物監視）」

## 優先度の所感（覆してよい）

| 項目 | 費用対効果 | 理由 |
|---|---|---|
| 2. 編集→判定ループ | **高** | ほぼ既存部品の配線。dev 用途の体験が一番変わる |
| 1. backend 代役/中継 | **高**（relay）〜中（emulate） | relay は薄い。emulate は契約記録の設計が要る |
| 5. 開発者アダプタ | **高**（骨組み）〜中（網羅） | 1・2 の恒久的な配線面。ただし先行必須ではない（下記） |
| 3. 論理スクリーン | 中 | 部品完備、パッケージングのみ。CI 文脈が来たら着手 |
| 4. focus dossier | 中（価値は最大級、コストも大） | モデル呼び出し設計・キャッシュ鍵設計が本体。discussion の T に積むべき |

推奨: **2 を単体スパイクで先行**（1 プロジェクト分のハードコード配線で動かしてよい）→ スパイクが実際に必要とした設定項目から **5 の manifest 語彙を抽出**（実物からの一般化。先に語彙を設計しない）→ 1(relay) → 3 → 1(emulate) → 4。5→2 の依存は必須ではなく、逆向きに育てる方が語彙が現実に根を張る。ロードマップ（README）の Phase 1〜3 とは直交せず、Phase 2/3 の部品を先に単体で価値化する順序になっている。
